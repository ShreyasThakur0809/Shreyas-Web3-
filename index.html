Execution & Transactions

How instructions are bundled and executed differs significantly, impacting developers and users.

Solana

Transaction Model

A transaction is a bundle of one or more atomic instructions. All instructions succeed or fail together.

Instruction 1: Create a data account (PDA) for a user.
Instruction 2: Initialize data in that account.
Instruction 3: Transfer SOL to the new account for rent.

This atomicity allows for complex, multi-step operations in a single, secure transaction.

Ethereum

Transaction Object

A transaction is a single message from an EOA to another account, containing key fields.

  • To: Contract or EOA address.
  • Value: Amount of ETH to send.
  • Data: Encoded function call and arguments.
  • Gas Limit/Price: Fee parameters.
  • Nonce: Transaction counter.

The EVM executes the code based on the `data` field, modifying the contract's internal storage.

Example: Send SOL (Solana)

Show Code > 
import { useConnection, useWallet } from "@solana/wallet-adapter-react";
import { LAMPORTS_PER_SOL, PublicKey, SystemProgram, Transaction } from "@solana/web3.js";

function SendTokens() {
  const wallet = useWallet();
  const {connection} = useConnection();

  async function sendTokens() {
    const to = "RecipientPublicKeyHere"; // Replace with actual recipient
    const amount = 1; // Amount in SOL

    if (!wallet.publicKey) return;

    const transaction = new Transaction().add(
      SystemProgram.transfer({
        fromPubkey: wallet.publicKey,
        toPubkey: new PublicKey(to),
        lamports: amount * LAMPORTS_PER_SOL,
      })
    );
    await wallet.sendTransaction(transaction, connection);
    console.log("Sent " + amount + " SOL to " + to);
  }
  return <button onClick={() => sendTokens()}>Send SOL</button>;
}

Example: Sign Message (Solana)

Show Code > 
import { useWallet } from '@solana/wallet-adapter-react';
import { ed25519 } from '@noble/curves/ed25519';
import bs58 from 'bs58';

function SignMessage() {
  const { publicKey, signMessage } = useWallet();

  async function onClick() {
    if (!publicKey || !signMessage) return;
    const message = "Hello Solana!";
    const encodedMessage = new TextEncoder().encode(message);
    const signature = await signMessage(encodedMessage);
    if (!ed25519.verify(signature, encodedMessage, publicKey.toBytes())) {
      throw new Error('Message signature invalid!');
    }
    console.log(`Signature: ${bs58.encode(signature)}`);
  }
  return <button onClick={() => onClick()}>Sign Message</button>;
}

async function sendEth() {
  if (typeof window.ethereum === 'undefined' || !window.ethereum.selectedAddress) {
    console.log("Ethereum wallet not connected.");
    return;
  }

  const provider = new ethers.providers.Web3Provider(window.ethereum);
  const signer = provider.getSigner();
  const toAddress = "0xRecipientAddressHere"; // Replace with actual recipient
  const amountEth = "0.001"; // Amount in ETH

  try {
    const tx = {
      to: toAddress,
      value: ethers.utils.parseEther(amountEth) // Converts ETH to Wei
    };
    const transactionResponse = await signer.sendTransaction(tx);
    console.log("Transaction sent:", transactionResponse.hash);
    await transactionResponse.wait(); // Wait for transaction to be mined
    console.log("Transaction confirmed!");
  } catch (error) {
    console.error("Failed to send ETH:", error);
  }
}
// To use: <button onclick="sendEth()">Send ETH</button>

Data Locations: `memory`, `stack`, `storage`

Solidity has distinct data locations with different characteristics and gas costs.

`storage` (Persistent)

Persistent data on the blockchain (state variables). Costly to write.

Show Code > 
contract StorageExample {
  uint[] public numbers; // Stored in storage

  function addNumber(uint _number) public {
    numbers.push(_number); // Modifies storage
  }
}

`memory` (Temporary, Function Scope)

Temporary data during function execution. Cheaper than storage. Used for function arguments (complex types) and local variables (complex types).

Show Code > 
function addStrings(string memory a, string memory b) public pure returns (string memory) {
  string memory result = string(abi.encodePacked(a, b)); // 'result' is in memory
  return result;
}

`stack` (Temporary, Small Values)

Very small, temporary values (function arguments for simple types, local variables for simple types). Cleared after function execution.

Show Code > 
function addNumbers(uint a, uint b) public pure returns (uint) {
  // 'a', 'b', and 'result' are stored on the stack
  uint result = a + b;
  return result;
}

Inheritance: Reusing Contract Logic

Solidity supports single inheritance, allowing contracts to inherit functionality from a base contract.

Example: `Vehicle` (Base) & `Car` (Child)

Show Code > 
// Vehicle.sol (Base Contract)
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract Vehicle {
  string public brand;

  constructor(string memory _brand) {
    brand = _brand;
  }

  function description() public pure virtual returns (string memory) {
    return "This is a vehicle";
  }
}

// Car.sol (Child Contract)
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "./Vehicle.sol"; // Import the Vehicle contract

contract Car is Vehicle { // Inherit from Vehicle
  uint public numberOfDoors;

  // Constructor that also initializes the parent contract
  constructor(string memory _brand, uint _numberOfDoors) Vehicle(_brand) {
    numberOfDoors = _numberOfDoors;
  }

  // Override the description function to make it more specific for a car
  function description() public pure override returns (string memory) {
    return "This is a car";
  }
}

Cross-Contract Invocation (CCI) & Interfaces

Contracts interact with each other via CCI. Interfaces define a contract's external functions without implementation, enabling safe interaction.

Example: `ParentContract` calling `MessageContract`

Show Code > 
// MessageContract.sol (The contract to be called)
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract MessageContract {
  string private message;

  function setMessage(string memory newMessage) public {
    message = newMessage;
  }

  function getMessage() public view returns (string memory) {
    return message;
  }
}

// IMessageContract.sol (Interface definition)
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IMessageContract {
  function setMessage(string memory newMessage) external;
  function getMessage() external view returns (string memory);
}

// ParentContract.sol (Calls MessageContract)
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "./IMessageContract.sol"; // Import the interface

contract ParentContract {
  address private contractAddress;

  constructor(address _contractAddress) {
    contractAddress = _contractAddress;
  }

  function setMessageFromOtherContract(string memory newMessage) public {
    // Call the external contract via its interface
    IMessageContract(contractAddress).setMessage(newMessage);
  }

  function getMessageFromOtherContract() public view returns (string memory) {
    // Get message from external contract via its interface
    return IMessageContract(contractAddress).getMessage();
  }
}

üíé ERC-20 & Tokens

Understanding fungible tokens on Ethereum: ERC-20 standard, how balances are managed, and common token operations.

What is a Token?

A fungible token is a type of currency on the blockchain with specific properties.

  • It is a type of currency.
  • It has an owner/issuer/minter.
  • There is a mapping of which address owns how much of the token.
  • Addresses should be allowed to transfer tokens to other addresses.

Where are Balances Stored?

Native ETH balances are part of the EOA/Contract account. Custom token balances are stored within the token contract's storage.

Conceptual Diagram: Balances

Ethereum World State

Address (EOA)

nonce, balance (ETH)

Address (Contract)

nonce, balance (ETH), storage hash, code hash

Custom token balances are in Contract Storage

Custom Tokens: Ethereum vs. Solana

The architecture for custom tokens differs significantly between chains.

  • **Ethereum:** Every custom token (e.g., USDC, USDT) is a **new, independently deployed smart contract**.
  • **Solana:** All tokens are **instances managed by a single, canonical SPL Token Program**.

ERC-20 Standard Specifications

The ERC-20 standard defines a set of functions and events for fungible tokens to ensure interoperability.

  • `totalSupply()`: Total number of tokens in circulation.
  • `balanceOf(address account)`: Balance of tokens held by a specific address.
  • `transfer(address recipient, uint256 amount)`: Transfers tokens from caller's account to recipient.
  • `allowance(address owner, address spender)`: Amount spender is allowed to transfer on behalf of owner.
  • `approve(address spender, uint256 amount)`: Approves spender to transfer tokens on caller's behalf.
  • `transferFrom(address from, address to, uint256 amount)`: Transfers tokens from `from` to `to` by an approved `spender`.

Note: The ERC-20 standard does NOT define minting or burning logic; these are added by the token issuer.

Example: Basic Token Contract (ERC-20-like)

A simplified token contract demonstrating core functionalities including minting and burning.

Show Code > 
contract Token {
  string public name;
  uint public supply = 0;
  address public owner;
  mapping(address => uint) public balances; // ERC-20 like balance mapping

  constructor(string memory _name) {
    name = _name;
    owner = msg.sender;
  }

  function mintTo(address to, uint amount) public { // Simplified minting logic
    require(msg.sender == owner, "Only owner can mint");
    balances[to] += amount;
    supply += amount;
  }

  function transfer(address to, uint amount) public { // ERC-20 transfer
    require(balances[msg.sender] >= amount, "Insufficient balance");
    balances[msg.sender] -= amount;
    balances[to] += amount;
  }

  function burn(uint amount) public { // Simplified burning logic
    require(balances[msg.sender] >= amount, "Insufficient balance to burn");
    balances[msg.sender] -= amount;
    supply -= amount;
  }
}

`approve` & `transferFrom` (ERC-20 Allowance)

This mechanism allows a third party (spender) to move tokens on behalf of an owner, essential for dApp interactions.

Show Code > 
contract TokenWithAllowance {
  mapping(address => uint) public balances;
  mapping(address => mapping(address => uint)) public allowance; // owner => spender => amount

  function approve(address spender, uint amount) public {
    allowance[msg.sender][spender] = amount;
    // emit Approval(msg.sender, spender, amount); // ERC-20 Approval event
  }

  function transferFrom(address from, address to, uint amount) public {
    require(balances[from] >= amount, "Insufficient balance");
    require(allowance[from][msg.sender] >= amount, "Allowance exceeded");

    balances[from] -= amount;
    balances[to] += amount;
    allowance[from][msg.sender] -= amount; // Deduct from allowance
  }
}

How to Mint/Launch a Token?

Common methods for distributing newly created tokens:

  • **Initial Coin Offering (ICO):** Selling tokens to raise capital.
  • **Airdrop:** Distributing tokens for free to a large number of addresses.
  • **Fair Launch:** No pre-sale or private allocation; tokens distributed transparently.
  • **Token Vesting / Token Unlocking:** Releasing tokens over time (e.g., for team members).
